# Introduction

In bug bounty hunting, many findings appear insignificant when viewed individually.

An HTML injection that only affects an email template may be classified as a low-severity issue. Likewise, a missing rate limit on a signup endpoint is often considered a spam prevention problem rather than a security vulnerability.

However, combining multiple low-impact issues can dramatically increase the overall risk.

This write-up explains how I chained an **Email HTML Injection** with a **Missing Rate Limit** on a signup endpoint to demonstrate how an application's own email infrastructure could be abused to send convincing phishing emails at scale.

> **Disclaimer:** This research was conducted under an authorized vulnerability disclosure or bug bounty program. The testing was limited to my own accounts and infrastructure. No unauthorized phishing campaign or mass emailing was performed.

---

# Target Overview

While assessing the authentication flow of the target application, I analyzed the user registration process.

The signup form requested the following information:

- Name
- Email Address
- Password

After successful registration, the application automatically sent a verification or welcome email to the supplied email address.

This raised an important question:

> **How is user-controlled input handled before being inserted into the HTML email template?**

---

# Vulnerability 1 — HTML Injection in Email Templates

I intercepted the registration request using Burp Suite and modified the **Name** parameter before forwarding it.

Example test payload:

```html
<font color="red">SECURITY ALERT: Unauthorized Login Attempt</font>
```

After completing the registration using my own email address, the automated email arrived.

Instead of displaying the payload as plain text, the HTML was rendered by the email client.

This indicated that user-controlled input was being inserted into the email template without proper output encoding.

# Demonstrating the Risk

To further demonstrate the impact, I tested whether additional HTML elements were rendered within the email template.

The application accepted HTML formatting and reflected it in the email body, allowing user-supplied content to influence the appearance of official emails.

This demonstrated that the email template did not properly encode user-controlled HTML before rendering.

# Vulnerability 2 — Missing Rate Limiting

Next, I evaluated whether the signup endpoint contained any protections against automated abuse.

Using Burp Suite Intruder, I replayed the registration request multiple times using my own controlled email address.

The server consistently responded with:

```
HTTP/1.1 200 OK
```

Observed protections:

- No IP-based throttling
- No request rate limiting
- No email-based cooldown
- No CAPTCHA challenge

Within seconds, multiple legitimate emails arrived in my test inbox.

# Chaining the Vulnerabilities

Individually, each issue had limited impact.

Combined, they significantly increased the potential risk.

An attacker could:

1. Automate requests to the registration endpoint.
2. Insert misleading HTML into user-controlled fields reflected in emails.
3. Trigger large numbers of legitimate emails generated by the application's infrastructure.

Because these emails originate from the organization's legitimate mail servers, they benefit from the sender's normal email authentication configuration (such as SPF, DKIM, and DMARC), increasing the likelihood of successful delivery.

The result is an abuse of trusted infrastructure to distribute misleading or deceptive content.

---

# Potential Impact

Successful exploitation could enable scenarios such as:

- Email content spoofing
- Abuse of trusted branding
- Large-scale unsolicited email delivery
- Increased phishing credibility
- Reputation damage
- Potential degradation of email deliverability if abuse becomes widespread

The severity arises from the interaction between multiple weaknesses rather than either issue alone.

# Recommended Remediation

## 1. Context-Aware Output Encoding

All user-controlled input included within HTML email templates should be properly encoded before rendering.

For example:

```html
<a>
```

should be rendered as:

```html
&lt;a&gt;
```

when displayed as text.

---

## 2. Rate Limiting

Implement rate limits on endpoints that trigger transactional emails.

Examples include:

- Per IP address
- Per email address
- Per account
- Per device fingerprint

---

## 3. Bot Protection

Add automated abuse protections such as:

- CAPTCHA
- Behavioral analysis
- Request throttling
- Progressive verification

---

## 4. Email Template Validation

Avoid inserting raw HTML from user-controlled fields into email templates.

Where formatting is required, use strict allowlists and sanitize input before rendering.

---

# Lessons Learned

- Low-severity issues should not always be evaluated independently.
- Chained vulnerabilities often produce significantly greater impact.
- Email infrastructure deserves the same security attention as web applications.
- Rate limiting is an important security control, not merely a spam prevention feature.
- Manual testing frequently uncovers attack paths that automated scanners overlook.

# Conclusion

Security vulnerabilities should rarely be viewed in isolation.

An HTML injection inside an email template might initially appear to be a minor issue, while a missing rate limit could be dismissed as operational abuse.

When combined, however, these weaknesses can significantly increase the risk by allowing trusted infrastructure to be abused in unintended ways.

This research reinforces an important lesson in bug bounty hunting:

> **Always look for how multiple findings interact—not just how they behave individually.**
